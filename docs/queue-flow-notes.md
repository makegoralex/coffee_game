# Заметки по реализации очереди заказов (для будущего себя)

Дата: 2026-03-12

## Что уже сделано

В проекте реализован базовый поэтапный цикл обработки клиента/заказа:

1. Клиенты спавнятся по таймеру из `customerFlowPerMinute`.
2. Клиенты попадают в `customerQueue`.
3. Из очереди создаются `Order` и кладутся в `orderQueue`.
4. Одновременно варится только **один** заказ (`brewingOrderId`).
5. После завершения таймера заказ переходит в `readyOrderIds`.
6. Деньги начисляются только на шаге выдачи готового заказа (кнопка `Выдать готовый заказ`).

## Где это находится

- Типы состояния: `src/shared/types/state.ts`
- Инициализация: `src/game/state/create-initial-state.ts`
- Игровой цикл: `src/game/app/game-app.ts`
- UI-индикаторы очереди/варки/готовности: `src/main.ts`

## Важные договорённости

- Статусы клиента: `waiting -> ordering -> brewing -> served -> left`.
- `served` выставляется **только при фактической выдаче** заказа.
- Пока заказ просто "готов к выдаче", клиент остаётся в `brewing` (в рамках текущей упрощённой модели).
- Терпение не тикает для клиентов со статусом `served`/`left`.

## Что легко перепутать потом

- `sellCoffee()` сейчас не "мгновенная продажа", а вызов выдачи готового заказа.
- `manualSaleIncome` пока остаётся в состоянии как legacy-поле, но не является основным источником дохода.
- В `CafeState` одновременно живут старые `services` и новый упрощённый поток заказов — это технический долг и точка для рефакторинга.

## Что сделать следующим шагом

1. Либо полностью удалить legacy `services`/`WorkerTask`,
   либо снова интегрировать их в новый order-flow, чтобы не было двух конкурирующих моделей.
2. Добавить отдельный статус клиента `ready_for_pickup`, чтобы убрать двусмысленность текущего `brewing`.
3. Добавить unit-тесты на переходы статусов и начисление денег только на handoff.
4. Сделать отдельный UI-блок с ID/прогрессом активного заказа (сейчас показывается только `orderId` или `—`).
