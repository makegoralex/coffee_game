# Service Loop Notes (GameApp)

Документ фиксирует, что уже реализовано в пошаговом цикле кофейни, **где это лежит** и **зачем сделано именно так**.

## Что реализовано

В `GameApp` реализован последовательный цикл:
1. спавн посетителей;
2. постановка в очередь;
3. создание заказа на кассе;
4. прохождение заказа по станциям;
5. приготовление и выдача;
6. уход клиента при успешной выдаче или потеря клиента по терпению/переполнению.

Главный принцип экономики: **деньги начисляются только после успешной выдачи заказа** на финальной станции (`pickup`).

## Где что находится

### 1) Типы состояния
Файл: `src/shared/types/state.ts`

Добавлены/расширены сущности:
- `Order` — заказ клиента и текущая станция обработки;
- `WorkerTask` — активная задача станции (таймер выполнения);
- `QueueState` — очередь клиентов перед кассой;
- `ServiceState` — очередь заказов и активная задача конкретной станции;
- `CafeState` расширен полями:
  - `activeOrders`,
  - `customerQueue`,
  - `services`,
  - `spawnRemainder`.

Зачем: отделить очередь клиентов от очередей по станциям и явно хранить прогресс заказа по pipeline.

### 2) Инициализация
Файл: `src/game/state/create-initial-state.ts`

Инициализация подтягивается из конфигов:
- `economy.json`:
  - `baseAverageCheck`,
  - `baseCustomerFlowPerMinute`;
- `stations.json`:
  - список станций,
  - `baseProcessingTimeSec`,
  - `baseQueueSize`.

Также поднимаются стартовые `services` и `customerQueue`.

Зачем: чтобы баланс регулировался данными, а не хардкодом в логике.

### 3) Игровой цикл
Файл: `src/game/app/game-app.ts`

Ключевые шаги в `tick(deltaSeconds)`:
- `updatePatience` — увеличивает ожидание клиента, теряет клиента при таймауте;
- `spawnCustomers` — спавнит по `customerFlowPerMinute` с дробным остатком `spawnRemainder`;
- `feedCashierQueueFromVisitors` — переносит клиента из общей очереди в заказ на кассе;
- `processServices` — двигает заказы по станциям `cashier -> espresso_machine -> pickup`.

Завершение заказа:
- `completeOrder` удаляет заказ/клиента и только здесь начисляет `wallet.soft`.

Потеря клиента:
- `loseCustomer` удаляет клиента из всех очередей, активных задач и связанных заказов.

Зачем: получить детерминированный step-by-step loop, прозрачный для отладки и будущего UI.

### 4) События
Файл: `src/game/events/event-bus.ts`

Добавлены доменные события:
- `customer.spawned`,
- `customer.lost`,
- `order.created`,
- `order.completed`.

Зачем: UI/аналитика могут подписываться на жизненный цикл клиента и заказа без прямой связки с `GameApp`.

## Важные ограничения текущей реализации

1. Pipeline станций сейчас фиксирован константой в `GameApp`:
   - `cashier -> espresso_machine -> pickup`.
2. `sellCoffee()` больше не «печатает деньги», а только триггерит обработку сервиса.
3. `computeOfflineIncome()` возвращает `0` — оффлайн доход в текущем loop отключён.

## Что проверять при дальнейших изменениях

1. Не появилось ли обходных путей начисления денег вне `completeOrder`.
2. Совпадают ли queue limits с `stations.json`.
3. Не ломается ли удаление клиента из:
   - `activeCustomers`,
   - `customerQueue`,
   - `activeOrders`,
   - `services[].queuedOrderIds`,
   - `services[].activeTask`.
4. Не меняется ли порядок шагов в `tick`, иначе легко получить гонки/неконсистентность.

## Точка входа для будущей доработки

Если нужно расширять loop (бариста, несколько линий, приоритеты):
1. сначала расширять типы в `state.ts`;
2. затем инициализацию в `create-initial-state.ts`;
3. только после этого логику в `game-app.ts`;
4. обязательно добавлять доменные события в `event-bus.ts`.
